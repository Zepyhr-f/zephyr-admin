# 后端工程结构设计

> 本文档规定 Zephyr Admin 后端的 Maven 多模块目录结构、各模块职责边界、代码分层约定，以及核心优化方案的落地位置，是所有后端开发者的**行为准则**。

---

## 1. 总体架构分层

```
基础设施层 (zephyr-base)
    ↑ 被所有层依赖
公共服务层 (zephyr-common)
    ↑ 被业务层调用
业务微服务层 (zephyr-biz)
    ↑ 通过网关对外暴露
网关层 (zephyr-gateway)
    ↑ 对外唯一入口
```

---

## 2. Maven 多模块目录结构

```
zephyr-server/
├── pom.xml                          # 根 POM：继承关系 + BOM 引入
│
├── zephyr-bom/                      # ① BOM 模块：统一锁定所有三方依赖版本
│   └── pom.xml
│
├── zephyr-base/                     # ② 基础设施层：纯技术能力封装，无业务逻辑
│   ├── zephyr-core/                 #    核心工具：R<T> 响应体、BaseEntity、异常体系
│   └── zephyr-starter/              #    能力 Starter：JWT、MyBatis-Plus、Redis 封装
│       ├── zephyr-starter-jwt/
│       ├── zephyr-starter-mp/       #    包含 MetaObjectHandler（审计字段自动填充）
│       └── zephyr-starter-redis/
│
├── zephyr-common/                   # ③ 公共服务层：通用业务服务
│   ├── zephyr-auth/                 #    认证授权（登录、Token 发行、双令牌刷新）
│   ├── zephyr-system/               #    系统管理（用户/角色/菜单/字典/部门 RBAC）
│   └── zephyr-ops/                  #    运维监控（健康、日志、系统信息）
│
├── zephyr-biz/                      # ④ 业务微服务层（按业务域独立扩展）
│   ├── zephyr-audit/                #    审计日志服务
│   └── zephyr-ai/                   #    AI 助手服务（Spring AI + Milvus + 语义缓存）
│
├── zephyr-gateway/                  # ⑤ 网关层：唯一对外入口
│   └── src/
│       └── main/java/
│           ├── filter/
│           │   └── AuthFilter.java  #    JWT 鉴权全局过滤器
│           └── config/
│               └── RouteConfig.java #    路由配置（从 Nacos 动态加载）
│
└── scripts/
    ├── z-server.sh                  # 后端统一管理脚本
    └── docker-compose.yml           # 本地依赖服务配置
```

## 3. 各模块职责详解

### 3.1 `zephyr-bom` — 依赖版本管理

**职责**：在一个地方统一锁定所有三方库版本，子模块引用依赖时**只写 groupId + artifactId，不写 version**。

**核心内容**：
- `spring-boot-dependencies`（Spring Boot BOM）
- `spring-cloud-dependencies`（Spring Cloud BOM）
- `mybatis-plus-bom`
- `mapstruct`、`lombok`、`flyway` 等固定版本

---

### 3.2 `zephyr-core` — 核心工具

| 组件 | 说明 |
|---|---|
| `R<T>` | **统一响应体**，所有接口返回此结构：`{ code, success, data, msg }` |
| `ResultCode` | 全局状态码枚举（200=成功，401=未认证，403=无权限，500=系统错误） |
| `BaseEntity` | 所有业务表基类：`id`、`createTime`、`updateTime`、`createUser`、`updateUser`、`deleted` |
| `BusinessException` | 业务异常基类，携带 `ResultCode` |
| `GlobalExceptionHandler` | `@RestControllerAdvice`，统一捕获并返回 `R.fail()` |

---

### 3.3 `zephyr-starter-mp` — MyBatis-Plus 能力

**关键配置**：
- `MetaObjectHandler`：自动填充 `create_time`、`update_time`、`create_user`（从 `SecurityContext` 获取）
- `OptimisticLockerInnerInterceptor`：乐观锁插件，配合实体上的 `@Version` 防止并发写冲突
- `PaginationInnerInterceptor`：分页插件
- `PerformanceInterceptor`：慢 SQL 监控（开发/测试环境启用）
- 逻辑删除：全局配置 `logic-delete-field: deleted`，调用 `removeById` 自动变为 `UPDATE ... SET deleted=1`

---

### 3.4 `zephyr-starter-jwt` — 双令牌能力

| 组件 | 说明 |
|---|---|
| `JwtUtil` | JWT 生成/解析，支持 ES256（默认）和 HS256 双算法切换 |
| `TokenService` | 双令牌管理：发行 Access Token（30 min）+ Refresh Token（7 天，存 Redis）|
| `BlacklistService` | Token 黑名单：将失效 `jti` 写入 Redis，网关过滤时比对 |

**双令牌流程**：
```
POST /auth/login
  └─ 验证用户名/密码（BCrypt）
       └─ 生成 Access Token（Header 返回）
       └─ 生成 Refresh Token（Set-Cookie: HttpOnly; SameSite=Strict）
       └─ Refresh Token 存入 Redis（Key: refresh:{username}, TTL: 7天）

POST /auth/refresh
  └─ 读取 Cookie 中的 Refresh Token
       └─ 比对 Redis 中存储的 Refresh Token
       └─ 有效 → 颁发新 Access Token

POST /auth/logout
  └─ 将当前 Access Token 的 jti 写入 Redis 黑名单（TTL = 剩余有效期）
  └─ 删除 Redis 中的 Refresh Token
```

---

### 3.5 `zephyr-auth` — 认证授权服务

| 端点 | 方法 | 说明 |
|---|---|---|
| `/auth/login` | POST | 登录，返回 Access Token + 设置 Refresh Cookie |
| `/auth/refresh` | POST | 刷新 Access Token |
| `/auth/logout` | POST | 下线，双 Token 同步失效 |
| `/auth/info` | GET | 从 Token 查询当前用户完整信息、权限列表 |

---

### 3.6 `zephyr-gateway` — 网关层

**AuthFilter 核心逻辑**：
```
请求进入
  └─ 是否白名单？（/auth/login, /auth/refresh, /actuator/**）
       ├─ 是 → 直接放行
       └─ 否 → 读取 Authorization: Bearer <token>
                  └─ JWT 签名验证
                  └─ 比对 Redis 黑名单（jti）
                  └─ 通过 → 在 Header 注入 X-User-Id、X-Username、X-Role-Id → 转发
                  └─ 失败 → 返回 401
```

**设计原则**：网关只做 **Token 有效性** 校验，细粒度权限（`@PreAuthorize`）下沉到各业务服务，避免网关逻辑过重。

---

### 3.7 `zephyr-ai` — AI 服务（语义缓存架构）

```
用户提问
  └─ 问题向量化（Embedding）
       └─ 查询 Redis 语义缓存（Key: semantic:hash:<md5>）
            ├─ 命中 → 直接返回缓存结果（毫秒级）
            └─ 未命中 → 调用大模型（Spring AI Provider）
                        └─ 结果写入 Redis（TTL 可配置）
                        └─ 向量写入 Milvus（供相似度检索）
```

## 4. 代码分层约定

每个业务微服务内部分层如下：

```
src/main/java/com/zephyr/<module>/
├── controller/          # 接口层：只做参数校验和响应封装，无业务逻辑
├── service/             # 业务层：核心逻辑，调用 Mapper 和外部服务
│   └── impl/
├── mapper/              # 持久层：MyBatis-Plus Mapper 接口
├── entity/              # 数据库实体：继承 BaseEntity
├── dto/                 # 数据传输对象（请求 DTO + 响应 VO）
│   ├── request/         #   如 UserCreateRequest、UserUpdateRequest
│   └── response/        #   如 UserVO、PageResult<UserVO>
├── convert/             # MapStruct 转换器（Entity ↔ DTO）
└── config/              # 模块内配置类
```

**命名规范**：
| 文件类型 | 命名 | 示例 |
|---|---|---|
| Controller | `XxxController` | `UserController` |
| Service 接口 | `IXxxService` | `IUserService` |
| Service 实现 | `XxxServiceImpl` | `UserServiceImpl` |
| Mapper | `XxxMapper` | `UserMapper` |
| 实体 | `Xxx` | `SysUser` |
| 请求 DTO | `XxxRequest` | `UserCreateRequest` |
| 响应 VO | `XxxVO` | `UserVO` |
| MapStruct 转换 | `XxxConvert` | `UserConvert` |

## 5. API 文档规范（Smart-Doc）

使用 Smart-Doc，在 `pom.xml` 中配置 Maven 插件，基于标准 JavaDoc 自动生成文档：

```xml
<plugin>
    <groupId>com.ly.smart-doc</groupId>
    <artifactId>smart-doc-maven-plugin</artifactId>
    <version>3.x</version>
    <configuration>
        <configFile>./src/main/resources/smart-doc.json</configFile>
    </configuration>
</plugin>
```

**规范**：
- 所有 Controller 方法必须有 JavaDoc 注释（`/** */`），描述接口功能
- 所有 DTO 字段必须有 JavaDoc 注释，描述字段含义与约束
- CI 中执行 `mvn smart-doc:html` 自动发布文档，无需手动维护

## 6. 落地优先级

| 阶段 | 内容 | 文件 |
|---|---|---|
| **S1（地基）** | `zephyr-bom` 建立 + 目录分层搭建 | `pom.xml` |
| **S2（效率）** | MapStruct 接入 + 统一异常处理 + Smart-Doc | `zephyr-core`、`convert/` |
| **S3（性能）** | 虚拟线程开启 + HikariCP 调优 | `application.yml` |
| **S4（安全）** | 双令牌机制实现 + Redis 黑名单 | `zephyr-starter-jwt` |
| **S5（特色）** | Spring AI + 语义缓存 + Milvus 接入 | `zephyr-ai` |
