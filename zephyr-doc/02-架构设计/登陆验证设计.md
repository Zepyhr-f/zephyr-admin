# 后台管理系统认证与鉴权全链路架构方案（优化版）

---

# 一、整体架构

## 技术栈

- 后端：Java + Spring Boot + Spring Security
- 网关：Spring Cloud Gateway
- 注册中心：Nacos
- 数据库：MySQL
- 缓存：Redis
- 前端：React

---

## 核心链路

```
前端请求
  ↓
Access Token（Header）
  ↓
Gateway（统一鉴权）
  ↓
内部可信 Header
  ↓
业务服务（用户上下文 + 权限校验）
  ↓
数据访问（租户隔离）
```

# 二、Token 设计

## 2.1 Access Token（JWT）

### Header

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

### Payload

```json
{
  "sub": "userCode",
  "tid": "tenantCode",
  "iat": 1710000000,
  "exp": 1710007200,
  "jti": "唯一ID"
}
```

### 说明

| 字段 | 说明                  |
| ---- | --------------------- |
| sub  | 用户编码(userCode)    |
| tid  | 租户编码(tenantCode)  |
| iat  | 签发时间              |
| exp  | 过期时间（建议2小时） |
| jti  | Token唯一标识         |

---

## 2.2 Refresh Token

### 存储方式（必须）

- HttpOnly Cookie
- Secure = true
- SameSite = Strict
- Path = /auth/refresh （最小权限原则，仅刷新接口可见，其他接口不可见）

### Payload

```json
{
  "sub": "userCode",
  "tid": "tenantCode",
  "jti": "refreshTokenId",
  "exp": 7天
}
```

---

## 2.3 Redis 存储结构

```
key: refresh:{jti}
value: userCode + tenantCode
ttl: 7天
```

# 三、前端方案

## 3.1 请求头规范

```
Authorization: Bearer {accessToken}
X-Request-Id: uuid
X-Client-Type: web / app
```

---

## 3.2 Token 刷新流程

```
1. 请求返回 401
2. 调用 /auth/refresh（自动携带 Cookie）
3. 返回新的 Access Token
4. 重放原请求
```

### 注意

- 必须加锁（避免并发刷新）
- 刷新失败 → 强制退出登录

---

## 3.3 用户信息与菜单路由（两步走）

### 1. 获取用户信息与权限

- **阶段**：登录成功后首个请求
- **接口**：`GET /system/user/info`
- **作用**：获取用户的基本信息、关联角色和所有按钮级权限标识。
- **控制逻辑**：前端基于 `permissions` 控制页面内按钮的显示/隐藏。

```json
{
  "user": {
    "userCode": "123456",
    "username": "admin"
  },
  "roles": ["super_admin"],
  "permissions": ["system:user:add", "system:user:delete"]
}
```

### 2. 获取动态菜单与路由

- **阶段**：获取用户信息之后
- **接口**：`GET /system/menu/tree`
- **作用**：根据用户的角色权限，后端返回经过过滤的菜单树与路由配置。
- **控制逻辑**：动态路由前端只认后端返回的数据进行挂载，不做本地路由的比对和硬编码维护。

```json
[
  {
    "path": "/system",
    "component": "Layout",
    "name": "System",
    "meta": { "title": "系统管理", "icon": "setting" },
    "children": [
      {
        "path": "user",
        "component": "system/user/index",
        "name": "User",
        "meta": { "title": "用户管理" }
      }
    ]
  }
]
```

# 四、网关设计

## 4.1 鉴权流程

- 校验 JWT 签名
- 校验 exp
- 校验 jti（可选）

---

## 4.2 Header 透传规范（必须统一）

```
X-User-Code
X-Tenant-Code
X-Timestamp
X-Gateway-Sign
```

---

## 4.3 签名机制（核心）

```
sign = HMACSHA256(
    userCode + tenantCode + timestamp,
    gateway-secret
)
```

---

## 4.4 安全策略

- 后端服务必须校验签名
- 校验 timestamp（防重放攻击）
- 内网服务禁止直接暴露

# 五、后端服务设计

## 5.1 用户上下文模型

```java
public class UserContext {
    private String userCode;
    private String tenantCode;
    private String username;
    private List<String> roles;
    private List<String> permissions; // 新增：权限标识集合
}
```

---

## 5.2 上下文处理流程

```
请求进入
  ↓
拦截器解析 Header (提取 userCode, tenantCode)
  ↓
校验签名
  ↓
本地服务从 Redis 提取用户详情 (装载 permissions)
  ↓
组装并写入 TransmittableThreadLocal (UserContext)
  ↓
业务使用 (接口权限校验直接从 UserContext 取，不用每次查库/缓存)
  ↓
请求结束清理
```

---

## 5.3 缓存设计

```
key: user:info:{userCode}
value: 用户信息JSON
ttl: 3600 + 随机(0~300)
```

# 六、权限模型设计

## 6.1 基于现存数据库模式的 RBAC 模型

系统基于标准 RBAC 并针对多租户及复杂组织架构进行了扩展。所有核心表均通过全局唯一的字符串编码（`code`）关联而非自增主键，核心流转图如下：

```
                 ┌──────────┐
                 │  租户表   │ (sys_tenant)
                 └────┬─────┘
                      │ tenant_code 隔离全局
      ┌───────────────┴────────────────┐
      │                                │
┌─────┴─────┐                    ┌─────┴─────┐
│  部门表    │ (sys_dept)         │  菜单表    │ (sys_menu, 包含菜单/按钮/API权限标识perms)
└─────┬─────┘                    └─────┬─────┘
      │                                │
      │(role_dept 数据权限)             │(role_menu 功能权限)
      │                                │
┌─────┴─────┐   (post_role 关联)       ┌─────┴─────┐
│  岗位表    ├───────────────────────── ┤  角色表    │ (sys_role)
└─────┬─────┘                          └─────┬─────┘
      │                                      │
      │(user_post 关联)                       │(user_role 直接关联)
      └─────────────────┬────────────────────┘
                        │
                   ┌────┴─────┐
                   │  用户表   │ (sys_user)
                   └──────────┘
```

**核心设计理念：**
1. **统一业务编码关联 (Code-based)**: 各表（如 `sys_user_role` 等）使用 `user_code`, `role_code`, `dept_code`, `menu_code` 进行物理与逻辑上的解耦关联，增强可读性并方便数据迁移。
2. **强制多租户隔离**: 核心业务表强制依赖 `tenant_code` 列，配合 `xxx_code` + `if_deleted` 形成租户内业务逻辑唯一约束（Unique Key）。
3. **扩展维度的双效权限管理**:
   - **功能权限**：通过 `用户 → 角色 → 菜单/权限标识` 实现界面元素与后台 API 的控制。
   - **数据权限**：角色可以直接关联至部门（`sys_role_dept`），用于控制角色可见的数据层级范围。
   - **组织与岗位权限（核心扩展点）**：一个岗位可以关联多个角色（`post_role` 机制）。用户除了能被直接赋予角色（`user_role`）外，还可以兼任多个岗位，并**隐式继承其所属岗位上挂载的所有角色**。这极大地简化了员工入职、调岗和离职时的权限变更成本，实现“岗变权变”。

# 八、安全设计

## 8.1 防重放攻击

- timestamp + 签名
- 可选：jti + Redis

---

## 8.2 Token 黑名单

```
key: blacklist:{jti}
ttl: token剩余时间
```

---

## 8.3 登录策略

| 模式     | 实现               |
| -------- | ------------------ |
| 单点登录 | 覆盖 refresh token |
| 多端登录 | 多 jti 管理        |

---

## 8.4 登出策略

登出接口必须严格执行双端状态与凭证清空动作：
1. **拉黑 Access Token**：将当前请求 accessToken 的 `jti` 加入 Redis 黑名单。
2. **销毁 Refresh Token**：删除 Redis 中存储的 refreshToken 映射。
3. **清空 Cookie**：服务端在 Response 中写入指令清空客户端 Cookie。
4. **清空本地缓存**：前端配合清空本地存储中的系统状态。

# 九、避坑要点

- ❌ 不使用 sessionStorage
- ❌ 不透传完整用户对象
- ❌ 不信任客户端传入信息
- ❌ 不直接使用 ThreadLocal
- ✔ 使用 TransmittableThreadLocal
- ✔ Gateway 必须做签名
- ✔ Refresh Token 必须使用 Cookie

---

# 十、总结

本方案核心目标：

- 建立 前端 → 网关 → 服务 的信任链
- 实现 认证、鉴权、缓存、权限 的闭环
- 提供 可扩展的多租户支持
- 保证 安全性（防伪造、防重放、防泄漏）
