# 文档规范与模板

> 本文档用于统一 Zephyr 文档库的写作规范、结构模板与维护流程。  
> 它既是新文档模板，也是现有文档的治理基线。

## 1. 基本原则（必须遵守）

1. **单一事实源**：版本号、端口、环境要求如需在多处出现，必须以《[版本与环境矩阵](./00-版本与环境矩阵.md)》为准，并尽量用“引用”代替“重复描述”。
2. **目录与命名**：
   - 一级目录使用 `0X-类目名称/`（已存在的目录保持兼容，新增必须遵循）。
   - 文件命名清晰可检索，避免“杂项/临时/最终版/新建文本文档”。
3. **README 责任**：每个一级/二级目录必须包含 `README.md`，用于：
   - 说明该目录范围
   - 列出文件清单（可用表格）
   - 提供修改指引与入口链接
4. **链接零死链**：任何 README 或导航页出现死链，视为文档质量问题，应在合入前修复。
5. **真实内容优先**：README 和导航只能描述仓库中真实存在的目录与文件；未来规划统一放到“规划中 / 待建设”区块，不直接链接不存在的路径。
6. **稳定优先于重命名**：默认先统一标题、README、元信息、引用与职责边界；除非必要，不做大规模物理重命名。

---

## 2. 文档分层与职责

| 类型 | 作用 | 典型位置 | 要求 |
|---|---|---|---|
| 目录入口 README | 提供范围、文件清单、关键词索引、维护动作 | 各级目录 | 必须存在 |
| SSOT 文档 | 承担唯一口径 | 版本矩阵、鉴权链路、数据库执行顺序等 | 必须显式标注 |
| 主题设计文档 | 描述某一设计主题的完整方案 | 架构设计、功能设计 | 结构完整、引用清晰 |
| 操作指南 | 描述如何启动、联调、排障、发版 | 快速开始、运维交付 | 面向执行 |
| 补充沉淀 | 记录专项说明、踩坑与经验 | 技术分享 | 不与 SSOT 冲突 |

## 3. 文档元信息头（关键文档必填）

> 关键文档包括：快速开始、架构设计、认证鉴权、数据库、开发规范、上线/运维手册等。

推荐在文档标题后增加以下表格：

```md
## 文档元信息
| 字段 | 值 |
|---|---|
| 状态 | Draft / Reviewed / Approved / Deprecated |
| Owner | 姓名或团队 |
| Reviewer | （可选） |
| 最后更新 | YYYY-MM-DD |
| 适用范围 | 前端/后端/全栈/某模块 |
| 对应代码版本 | commit hash / tag（可选） |
| 关联文档 | 相对链接列表 |
```

## 4. 常用文档结构模板

### 4.1 快速开始 / 环境搭建类（推荐结构）

```md
# 标题

## 文档元信息
...

## 1. 目标与范围
## 2. 前置条件（版本与权限）
## 3. 一键启动/停止（命令 + 说明）
## 4. 启动后验证（健康检查/页面/接口）
## 5. 常见问题排障（按错误现象→原因→解决）
## 6. 变更影响面（哪些改动必须同步更新本文档）
```

### 4.2 架构设计类（推荐结构）

```md
# 标题

## 文档元信息
...

## 1. 背景与目标（为什么这样设计）
## 2. 系统边界（C4 Context）
## 3. 服务/模块划分（C4 Container）
## 4. 核心链路（时序图：登录/鉴权/权限缓存/审计）
## 5. 数据模型（关键表/缓存 Key/消息主题）
## 6. 非功能性（性能/可用性/安全/可观测）
## 7. 风险与权衡（trade-offs）
```

### 4.3 功能设计类（推荐结构）

```md
## 1. 功能目标
## 2. 角色与权限点（RBAC/数据权限）
## 3. 业务流程（流程图/状态机）
## 4. 页面与交互（列表/筛选/按钮/校验规则）
## 5. API 清单（路径/入参/返回/错误码）
## 6. 表结构与字段（含索引）
## 7. 验收标准（可测试）
```

```md
- 是否存在“规划中 / 待建设”内容
```

## 6. 图表与排版规范（建议）

1. 图表优先使用 **Mermaid**（版本可控、可 diff、易维护）。
2. 所有命令块使用 ```bash；配置示例使用 ```yaml / ```json。
3. 使用 Markdown Callout（如 `> [!IMPORTANT]`）标注强制事项、风险提示、最佳实践。

---

## 7. 文档变更触发器（强制）

出现以下变更，必须同步更新对应文档：
- 版本矩阵变化（JDK/Node/pnpm/关键依赖升级）
- API 路径、返回体、错误码发生变化
- 数据库表结构、字段、索引发生变化
- 权限模型、菜单路由、鉴权策略发生变化
- 一键脚本/启动方式发生变化

---

## 8. 统一维护约束

1. 新增文档时，先更新所属目录 `README.md`，再补正文。
2. 同一主题只能有一个主文档；引用文档必须说明“以某文档为准”。
3. 规划中的内容必须单独标注，不得伪装成已存在目录或已完成功能。
4. 关键文档的 `最后更新` 字段必须在实质修改时同步更新。
