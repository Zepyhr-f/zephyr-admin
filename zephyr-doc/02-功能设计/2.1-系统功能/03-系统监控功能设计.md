# 系统监控功能详细设计

本文档详细描述了 Zephyr 系统监控模块的具体实现，包括对服务器硬件、中间件状态的实时监测以及定时任务的在线管理。

## 1. 服务监控 (Server Monitoring)
实时观察物理服务器或容器的运行状态，为性能调优和故障排查提供数据支持。

- **实现机制**：后端利用 OSHI (Operating System and Hardware Information) 库获取系统指标。
- **展示字段**：
  - **CPU 信息**：核心数、用户使用率、系统使用率、当前空闲率。
  - **内存信息**：总内存、已用内存、剩余内存、使用率。
  - **服务器信息**：服务器名称、操作系统、服务器 IP、系统架构。
  - **JVM 信息**：Java 版本、安装路径、启动时间、运行时长、占用内存、最大可用内存。
  - **磁盘状态**：盘符路径、文件系统、总大小、可用大小、已用大小、资源利用率。
- **页面交互**：卡片式布局，仪表盘或进度条展示负载情况，数据实时/定时刷新。

## 2. 缓存监控 (Cache Monitoring)
监控 Redis 中间件的运行效率与资源占用。

- **实现机制**：后端执行 Redis `INFO` 命令并解析返回结果。
- **核心指标**：
  - **版本信息**：Redis 版本、运行模式（单机/集群）、端口。
  - **运行性能**：已运行天数、连接客户端数。
  - **内存储备**：Redis 占用内存、内存峰值、内存碎片率。
  - **执行效率**：每秒执行命令数、Key 命中数、Key 丢失数。
- **功能特性**：
  - **命令统计**：展示各类 Redis 命令的调用频次占比（饼图）。
  - **Key 管理**：展示当前数据库的 Key 总数。

## 3. 数据源监控 (Data Source Monitoring)
监控数据库连接池状态，识别潜在的慢查询风险。

- **涉及组件**：Druid Monitor (内置)。
- **展示内容**：
  - **连接池状态**：当前活跃连接数、最大活跃数、空闲连接数。
  - **SQL 统计**：统计 SQL 执行次数、平均耗时、最长耗时。
  - **慢 SQL 追踪**：记录超过阈值（如 1s）的 SQL 语句。
- **实现方式**：前端采用 `iframe` 嵌入 Druid 后台管理页面（`/druid/index.html`），后端配置 Druid 过滤器及账号权限。

## 4. 定时任务 (Job Scheduling)
实现在线配置、调度并追踪周期性业务逻辑。

- **涉及表**：`sys_job` (任务配置), `sys_job_log` (执行日志)
- **核心字段**：
  - `job_name`: 任务名称
  - `job_group`: 任务组名
  - `invoke_target`: 调用目标（Bean 名称或方法）
  - `cron_expression`: Cron 表达式
  - `status`: 任务状态（0正常 1暂停）
- **功能按钮**：
  - `执行一次`: 立即触发一次任务逻辑。
  - `状态切换`: 启动/停止定时任务的调度。
  - `调度日志`: 跳转至日志详情，查看任务执行耗时及成功/失败状态。

---

## 5. 交互规范
1. **数据实时性**：监控类数据采用“进入页面即加载”模式，并提供“手动刷新”按钮。不建议过度频繁的后端轮询。
2. **可视化表达**：对于 CPU/内存等百分比数据，优先使用 `Progress` (进度条) 或 `Gauge` (仪表盘) 组件。
3. **敏感操作校验**：定时任务的“执行一次”与“状态修改”需具备操作日志记录（OperLog），并进行二次确认交互。
