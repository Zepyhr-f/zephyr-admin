# 编程质量与 Clean Code 规范 (Java 版)

本规范旨在确保后端代码的可读性、健壮性与高性能。在开发过程中请严格遵守。

## 编程质量准则

### 验证信息与异常处理
- **防御式编程**：对外提供的接口必须对参数进行合法性校验。
- **业务异常**：禁止捕获异常后只打印日志而不抛出。应封装统一的 `BusinessException`，由全局异常处理器拦截并返回给前端友好的错误码。
- **自定义状态码**：使用 `ResultCode` 枚举管理所有 API 状态码，避免在代码中硬编码魔法数字。

### 线程安全与并发
- 避免在 Service 层使用成员变量存储状态（Service 默认为单例）。
- 使用线程池（ThreadPoolExecutor）而非手动创建线程。
- 任务执行必须指定名称，以便于排查 CPU 飙升或死锁问题。

### 保护现有逻辑
- 重构代码前必须确保已有业务逻辑不被破坏。
- 定量评估核心代码修改对上游服务的影响。

## Java Clean Code 准则

### 命名规范 (Naming)
- **遵循 Java 规范**：包名全小写，类名大驼峰（UpperCamelCase），方法/变量名小驼峰（lowerCamelCase），常量全大写下划线（UPPER_SNAKE_CASE）。
- **对象命名约定**：
    - `Entity`: 对应数据库表。
    - `DTO`: 数据传输对象。
    - `VO`: 展示层对象。
- **语义清晰**：命名应反映其业务含义，如 `findByUsername` 优于 `find1`。

### 聪明的注释 (Smart Comments)
- **代码自解释**：如果命名足够清晰，不要画蛇添足写注释。
- **Javadoc**：公共接口（Service 接口、Controller）必须编写 Javadoc，说明参数、返回值及可能的异常。
- **TODO 跟踪**：临时的 HACK 或待办事项必须标注 `// TODO: [Name] Description`。

### 函数与类设计
- **短小精悍**：单一函数尽量不要超过 50 行，超过则考虑拆分。
- **职责单一**：一个类只负责一件事（SRP）。
- **适当封装**：通过 `Lombok` 注解（`@Getter`, `@Setter`, `@Builder`）简化代码，但避免在 Entity 中过度使用 `@Data`（可能会触发循环依赖或性能问题，建议明确指定相关注解）。

### DRY (Don't Repeat Yourself)
- 提取工具类（Util）或基类（Base）来处理通用的日期、字符串、加解密逻辑。
- 业务逻辑复建采用 Service 注入方式，严禁 Copy-Paste 代码段。

### 日志记录规范 (Logging)
- **使用 Slf4j 注解**：通过 `@Slf4j` 进行日志记录。
- **日志级别**：
    - `ERROR`: 异常流程或故障，必须记录堆栈信息。
    - `WARN`: 潜在风险或业务限制提醒。
    - `INFO`: 核心业务路径的关键步骤。
    - `DEBUG`: 开发调试信息（生产环境通常关闭）。
- **禁止在循环中打印日志**。

## 测试规范

- **单元测试**：针对 Service 层的核心算法及逻辑编写 JUnit 测试。
- **断言（Assert）**：关键路径使用断言确保状态。
- **Mock 服务**：使用 Mockito 模拟 Mapper 层，确保单测不依赖真实数据库。
