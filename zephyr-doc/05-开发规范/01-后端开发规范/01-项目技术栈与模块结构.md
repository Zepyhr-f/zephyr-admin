# 项目技术栈与模块结构规范

## 技术栈

### 核心框架与环境
*   **Java 21 (LTS)**: 项目后端统一基线版本（如需兼容更低版本，请在《版本与环境矩阵》声明）。
*   **Spring Boot 3.x**: 核心应用框架（具体版本以《版本与环境矩阵》为准）。
*   **Spring Cloud 2023.0.x**: 微服务治理方案（以版本矩阵为准）。
*   **Spring Cloud Alibaba 2022.0.x**: 国内生态套件（Nacos/Sentinel/Seata），以版本矩阵为准。
*   **Maven**: 模块化依赖管理与构建工具。

### 数据持久化
*   **MySQL 8.0+**: 核心关系型数据库。
*   **MyBatis-Plus 3.x**: 基于 MyBatis 的增强工具，提供高效的 CRUD、自动填充与分页插件。
*   **Redis**: 分布式缓存与共享会话存储。

### 开发效率工具
*   **Lombok**: 通过注解消除冗长的 Java 代码（Getter, Setter, Slf4j 等）。
*   **MapStruct**: 高性能的对象属性映射工具（DTO <=> Entity）。
*   **Hutool**: Java 常用工具类库。

> [!IMPORTANT]
> 本文档仅描述“技术选型与结构规范”，**具体版本号**请统一以《00-项目简介/00-版本与环境矩阵.md》为准，避免多处重复导致口径不一致。

## 模块结构规范

Zephyr 后端采用 Maven 多模块架构，遵循“核心功能内聚、业务功能解耦”的原则：

```
zephyr-server/
├── zephyr-base/                # 基础基础设施模块 (Starter, Core Utils)
│   ├── zephyr-starter-core/
│   ├── zephyr-starter-mp/
│   ├── zephyr-starter-redis/
│   └── zephyr-starter-security/
├── zephyr-common/              # 系统核心模块
│   ├── zephyr-system/          # 系统管理核心 (用户、角色、菜单、部门)
│   ├── zephyr-auth/            # 统一认证授权模块
│   ├── zephyr-ops/             # 运维相关功能 (如：任务调度、日志监控)
│   ├── zephyr-support/         # 系统支撑组件 (如：代码生成、测试套件)
│   └── common-boot/            # 系统模块统一启动入口 (整合 System/Auth/Ops/Generator)
├── zephyr-gateway/             # 基于 Spring Cloud Gateway 的统一网关
└── zephyr-biz/                 # 业务功能模块
    ├── rule/                   # 业务规则引擎/模块
    └── biz-boot/               # 业务模块启动入口
```

### 业务微服务内部结构 (Standard Layering)
每个 `biz` 微服务通常遵循以下包结构：

```
com.zephyr.[module]/             # 根包名
├── controller/                  # RESTful API 控制层（仅处理请求路由与验证）
├── service/                     # 业务接口层
│   └── impl/                    # 业务实现层（包含事务管理与复杂业务逻辑）
├── mapper/                      # 数据持久化接口层 (DAO)
├── entity/                      # 数据库实体类（对应数据库表）
├── domain/                      # 领域对象
│   ├── dto/                     # Data Transfer Object（接收/传输数据）
│   └── vo/                      # View Object（返回给前端展示的对象）
└── util/                        # 模块内部工具类
```

> [!IMPORTANT]
> **绝对禁止**在 `Controller` 层编写业务逻辑。Controller 只应负责请求参数的基本验证（使用 `@Valid`）和调用 Service。事务控制（`@Transactional`）应统一加在 `Service` 实现类上。
