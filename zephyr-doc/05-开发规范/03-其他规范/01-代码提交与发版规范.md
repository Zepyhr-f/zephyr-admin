# 标准代码提交流程 (Code Submission SOP)

> **适用范围**：Zephyr 项目前后端全员
> **触发时机**：当用户要求“提交代码”、“commit”或“生成 PR”时，Agent 必须严格执行以下 SOP。

---

## Phase 1: 预检查 (Pre-Commit Checks)

在执行任何 Git 提交操作前，Agent 必须自主完成以下质量检查。如果检查失败，**立即停止**流程，向用户报告错误原因并尝试修复。

### 1.1 后端 (Java) 检查
- **触发条件**：修改了 `zephyr-server` 目录下 `src/main/java` 中的 `.java` 文件。
- **操作指令**：针对受影响的类运行单元测试。
    - **指令示例**：`mvn test -Dtest=相关类名`
    - **要求**：测试必须 100% 通过（Green）。

### 1.2 前端 (React) 检查
- **触发条件**：修改了 `zephyr-web` 目录下的 `.ts` 或 `.tsx` 文件。
- **操作指令**：使用 Biome 进行代码规范检查。
    - **指令**：`pnpm biome check --write --files-ignore-unknown=true [修改的文件路径]`
    - **要求**：检查并修复所有可自动修复的 Lint 错误。

---

## Phase 2: 构建 Commit Message

Agent 必须根据 `git diff` 的内容自主生成符合 **Angular 规范** 的提交说明。

### 2.1 格式要求
`<type>(<scope>): <subject>`

### 2.2 字段说明
- **Type (必须)**：
    - `feat`: 新功能
    - `fix`: 修补 Bug
    - `docs`: 文档变更
    - `style`: 格式调整（不影响逻辑）
    - `refactor`: 重构
    - `test`: 增加测试
    - `chore`: 构建过程或辅助工具的变动
- **Scope (可选)**：指出修改的模块（如 `auth`, `gateway`, `user-ui` 等）。
- **Subject (必须)**：简洁准确的中文描述。

### 2.3 示例
- `feat(gateway): 增加 JWT 令牌失效自动清理逻辑`
- `fix(user-ui): 修复分压插件在移动端显示错位的问题`
- `docs(spec): 更新后端开发规范中的异常处理章节`

---

## Phase 3: 执行提交与推送

1.  **提交代码**：
    - 执行 `git add .`
    - 执行 `git commit -m "[生成好的Commit Message]"`
2.  **用户确认**：
    - 向用户展示生成的 Commit Message。
    - 使用 `ask_user` 获取确认。
3.  **推送代码**：
    - 确认无误后执行 `git push`。

> [!CAUTION]
> **绝对禁止**在质量检查失败的情况下强制提交。
> **严禁**直接 push 到 `main` 分支，必须根据 [Git 工作流规范](file:///Users/zephyr/z-code/zephyr-admin/zephyr-doc/05-开发规范/01-后端开发规范/03-Git工作流与提交规范.md) 操作分支。
