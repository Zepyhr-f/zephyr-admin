# 05-开发规范

本模块旨在确立项目开发中的“技术红线”与协作共识。通过标准化的文档约束，确保每一行代码、每一张数据表、每一次提交都具备一致的高质量水准。

---

## 使用原则

1. **先读后做**：收到开发任务后，先根据任务类型定位到后端、前端或协作规范目录。
2. **提交前过 SOP**：执行 `commit`、发版或交付动作前，必须阅读并遵守 [01-代码提交与发版规范.md](./03-其他规范/01-代码提交与发版规范.md)。
3. **版本口径统一**：文档中的工具链或环境版本如与《版本与环境矩阵》冲突，以版本矩阵为准。

## 核心规范摘要

-   **数据库**：所有业务表必须继承 `BaseEntity`（含 5 大审计字段），物理删除被严格禁止。
-   **代码质量**：后端严禁吞掉异常，前端禁止滥用 `any`。所有 API 必须使用统一响应包装。
-   **提交流程**：触发 commit 操作前，前端必须通过 Biome 检查，后端必须通过关键类单测。

> [!NOTE]
> 本目录中的“版本号/环境要求”如与《00-项目简介/00-版本与环境矩阵.md》冲突，以版本矩阵为准；请优先更新版本矩阵，再同步更新相关规范文档。

## 维护规则
- 新增规范前，先判断它属于后端、前端还是通用协作，避免同一规则散落多个目录。
- 同一主题的规范优先保留一份主文档；其他目录只保留引用入口。
- 当规范影响提交、发版、测试或基础设施约束时，必须同步检查是否需要更新快速开始和架构文档。
