# 01-代码提交与发版规范

本文档约定了项目代码提交、版本标记与发版日志的标准流程。**AI 助手在收到任何涉及"提交"、"发版"或"推送"的指令时，必须严格按照此规范执行，不得省略任何步骤。**

---

## 1. 触发场景

当用户的任何指令中包含以下关键词时，自动进入发版流程：
- "提交任务"
- "发版"
- "提交并推送"

---

## 2. 发版执行流程（强制全量执行）

### 步骤 1：版本号推算
- 若用户**明确指定**了版本号（如 `v1.0.5`），则直接使用。
- 若用户**未指定**，AI 需主动读取最后一个 Git Tag，并将修订号（Patch 位）自增 1（如 `v1.0.3` → `v1.0.4`）。

### 步骤 2：生成版本日志（严禁跳过）
**每次提交前**，AI 必须：
1. 自动收集并概括 `zephyr-server`、`zephyr-web`、`zephyr-doc` 三个模块本次的核心变动。
2. 在 `zephyr-doc/08-更新日志/` 目录下生成一篇 Markdown 格式的版本日志，命名规则为 `v<版本号>-changelog.md`（如 `v1.0.4-changelog.md`）。

**版本日志模板**:
```markdown
# v<版本号> 更新日志

**发版时间**: YYYY-MM-DD

## 后端变更 (zephyr-server)
- ...

## 前端变更 (zephyr-web)
- ...

## 文档变更 (zephyr-doc)
- ...
```

### 步骤 3：对三个子模块执行标准化 Git 流程

依次针对 **`zephyr-server`**、**`zephyr-web`**、**`zephyr-doc`** 三个目录，运行以下命令：

```bash
# 暂存所有变更
git add .

# 提交（Commit message 规范）
git commit -m "chore(release): version <版本号> - <AI 总结本次变动概要>"

# 打版本标签
git tag v<版本号>

# 一次性推送代码与 Tag
git push origin HEAD && git push --tags
```

> **注意**：根目录（`z-manager` 层）**不需要**执行 Git 操作。

---

## 3. 日志排查规范

当系统出现错误需要 AI 协助排查时：
- **统一日志目录**：`/Users/zephyr/z-code/z-manager/logs/`
  - 后端日志：`backend.log`
  - 前端日志：`frontend.log`
- **禁止全文通读**：必须采用倒序策略读取最新日志，优先使用 `tail -n 150` 提取尾部日志块，严禁从头阅读导致上下文膨胀。
