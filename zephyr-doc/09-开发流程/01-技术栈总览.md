# 技术栈总览

> 本文档面向开发者，完整描述 Zephyr Admin 的全栈技术选型，涵盖前端、后端、基础设施三大层次，并说明每项技术的选型理由。

## 4. 技术选型决策说明

### 4.1 为什么选 React 而非 Vue？
- 生态更成熟，TypeScript 支持更完善
- 函数式编程范式，与 Hooks 天然契合
- 企业级管理系统有大量 React + Ant Design 最佳实践可参考

### 4.2 为什么用 Zustand 而非 Redux Toolkit？
- 大幅减少样板代码，API 极其简洁
- 支持 devtools、persist、immer 等中间件
- 管理系统状态复杂度适中，无需 Redux 全套方案

### 4.3 为什么用 TanStack Query？
- 接口数据缓存、自动重试、后台刷新开箱即用
- 彻底解决 useEffect + useState 管理接口状态的混乱问题
- 与 Zustand 分工明确：服务端状态用 Query，客户端 UI 状态用 Zustand

### 4.4 为什么选 Biome 而非 ESLint + Prettier？
- 单一工具，无需维护两套配置
- Rust 编写，速度是 ESLint 的 25x
- 规则集与 ESLint 高度兼容，迁移成本低

### 4.5 为什么后端用微服务而非单体？
- 各业务模块可独立部署和扩展
- 通过 Nacos 实现动态配置，无需重启服务
- 为后续 AML、AI 等独立业务模块预留扩展空间
